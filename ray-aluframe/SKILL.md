# ray-aluframe：铝型材框架设计

把“我想做个这样的架子”变成一份商家看得懂、用户装得起来、关键假设能追溯的方案。不要把“模型画出来了”误当成“可以安全制作”。

## 核心原则

1. **先定义外形和用途,再选型材。** “3030 承重 50 kg”不是完整条件;跨度、载荷位置、支撑方式和具体厂家型号都会改变结果。
2. **照片只提供结构意图,不提供可靠尺寸。** 没有已知参照物时不要从像素猜绝对尺寸。
3. **通用方案与采购方案分开。** 未选厂商时可以给结构和规格要求,但不得编造货号、连接件或价格。
4. **所有数字标明来源或假设。** 厂商数据附页面和查询日期;经验假设显式列出,不得混入已确认事实。
5. **受力计算是筛查,不是认证。** 铝型材本体通过初查,仍可能因连接件、底脚、面板、偏载或装配误差失效。
6. **变更必须局部重算。** 改尺寸后列出受影响构件、清单变化和新风险,不要无说明地整套重做。

## 工作流

### 1. 判断任务等级

先分三类:

- **概念级**:只有用途、参考图或大概尺寸。交付结构建议和待确认清单。
- **询价级**:尺寸、载荷、厂商/系列和使用环境明确。交付可发商家的下料与配件清单。
- **制作级**:还要求孔位、加工、装配和最终核对。必须有具体厂商资料,并保留商家/专业人员复核闸门。

以下任一情况进入**受限模式**:载人、吊装、悬空、楼梯/护栏、车辆关键结构、鱼缸等破坏后后果严重的承载物、机器人/主轴等高速运动设备、明显冲击或疲劳载荷。受限模式只整理需求、风险与专业复核问题,不输出“可直接制作”结论。

### 2. 做最小澄清

优先复用用户已经说过的信息。一次只追问会改变结构的缺口,不要把表单整张甩给小白。至少确认:

- 做什么、放在哪里、谁会接触它
- 长/宽/高是外尺寸、净空还是台面尺寸
- 总载荷、单层载荷、载荷是均布还是集中、是否振动/移动
- 是否要层板、门、护罩、脚轮、调平脚、墙面固定
- 厂商/购买渠道是否已定;未定时是否接受先做通用方案
- 最在意成本、刚度、轻便、外观还是以后改装

带轮结构还要确认满载总重、使用地面/门槛、移动频率、是否在载荷偏向一侧时推动、几只轮带刹车、把手位置和停放时的防倾倒方式。

可以给“方便理解的尺寸示例”,但不得把任意示例包装成免测量的采购捷径。用户必须确认外尺寸、净空和关键物体尺寸后才能进入询价级;否则保持概念级。

从图片还原时,先找一个用户能确认的已知尺寸;没有参照尺寸就只描述拓扑,不报绝对长度。完整提问与交付模板见 [workflow-and-deliverables.md](references/workflow-and-deliverables.md)。

### 3. 先画载荷路径

用一句话说明重量如何从物体传到层板、横梁、立柱和地面。再确定:

- 立柱数量与位置
- 长边/短边横梁方向
- 是否需要中撑、斜撑或背板抗侧摆
- 高宽比与防倾倒措施
- 脚轮或底脚如何把力传到地面

至少给一个**推荐方案**。只有当成本、轻便或可拆装存在明显取舍时,再给一个备选;不要堆三四个近似方案逃避判断。

### 4. 绑定真实产品数据

采购级方案必须为每种型材和连接件记录:

- 厂商、系列、货号、外形尺寸、槽型
- 每米重量、供货长度
- 两个方向的惯性参数;需要强度检查时还要截面模量和材料强度
- 适配螺母、螺栓、角码/内连接件
- 是否需要端面攻丝、通孔、沉孔或其他加工
- 来源页面和查询日期

优先使用厂商目录、官方产品页和官方计算工具。电商标题只能用于发现候选,不能作为承载参数来源。字段说明见 [design-data.md](references/design-data.md)。

首版附带的 [starter-profile-data.json](references/starter-profile-data.json) 只用于演示 20/30/40 系列之间的数据差异和启动筛选;它不是完整产品库,连接件也必须另查。

面向淘宝常见欧标类商品时,先读 [euro-slot-guide.md](references/euro-slot-guide.md) 和 [euro-slot-hardware.json](references/euro-slot-hardware.json)。商品标题写“欧标”“30 系列”或“8 系列”仍不够;必须同时核对型材外形、实际槽宽、中心孔、槽螺母螺纹、角码孔径和开槽面。默认优先在同一家商家配齐型材、连接件、螺母和加工,不要跨店凭外观混配。

### 5. 建设计清单并运行检查

复杂方案或任何询价级/制作级方案,把设计写成 JSON,字段按 [design-data.md](references/design-data.md),然后运行:

```bash
python3 <skill-dir>/scripts/check_frame.py design.json --report report.md
python3 <skill-dir>/scripts/render_frame_svg.py design.json --output preview.svg
python3 <skill-dir>/scripts/render_assembly_svg.py design.json --output assembly-preview.svg
python3 <skill-dir>/scripts/generate_construction_pdf.py design.json --output construction-report.pdf
```

构件密集导致标签遮挡时,加 `--labels none` 生成给用户确认的干净预览;完整构件编号仍以检查报告和设计清单为准。

脚本负责:

- 检查构件编号、坐标、重复杆件和悬空端点
- 检查连接件与构件引用是否完整
- 对明确的横梁做变形和可选强度初查
- 按原料长度、锯缝和端部修整生成下料组合
- 汇总型材、连接件、附件和加工项
- 判定当前是“草案”“待复核”还是“可询价”

预览图用于让小白核对构件位置和整体比例,不是加工图。生成后实际打开检查,确认没有漏杆、错层或标签遮挡;用户确认预览后再整理询价包。

用户希望“接近实际效果”时,在设计清单的 `visuals` 中补层板、背板/展示板和调节脚,再生成装配效果图。装配效果图用银色带槽型材、节点五金、板材和底脚表达材料关系;它不是照片,不得据此判断颜色、表面缺陷或连接件精确外形。效果图里的每个附件必须在采购清单中有对应条目。

用户需要类似 MayCAD 的正式文档时,生成 PDF 施工文档。PDF 应包含等轴结构图、按型材和长度合并的材料清单、原料切割组合、横梁初查、装配顺序和待商家确认事项。未绑定厂商的连接件、板材或加工不得伪装成已确定;在文档中明确标为待确认。生成后必须把每页转成图片并实际检查版面。

不要手工覆盖脚本报出的失败项。设计数据不全时,保留“未检查”,不要拿默认值伪装通过。

### 6. 输出用户真正能用的交付包

按这个顺序输出:

1. **一句话结论**:推荐什么结构,当前能否询价/制作
2. **已确认条件与假设**
3. **尺寸与结构图**:标明坐标方向、外尺寸、净空、层高
4. **下料表**:构件编号、型材货号、长度、数量、端部处理
5. **连接与加工表**:每个节点的连接件、数量、孔位/攻丝位置
6. **采购清单**:型材、连接件、附件、层板/面板,区分必选与可选
7. **装配顺序**:先基准框、再立柱、再中层、最后校方和复紧
8. **检查结果**:通过项、未检查项、阻断项
9. **发给商家的确认问题**:兼容性、加工、公差、现货和报价

连接件数量、型号和加工位置是正式交付物,不能只写“若干”。如果尚未选厂商,明确写“结构数量已确定,货号与加工待厂商绑定”。

### 7. 处理修改

保存一份上版设计清单。用户说“宽改成 1500”时:

1. 识别所有受影响的长边杆件、层板、载荷跨度和下料组合。
2. 只修改这些项及其相关节点。
3. 重新运行检查。
4. 输出“增加/删除/变更/未变化”四类差异。
5. 若原型材不再满足,解释原因并给出最小改动,优先考虑中撑、改变受力方向或升级关键横梁,而非全架盲目加粗。

## 禁止事项

- 不凭“2020/3030/4040”名称宣称承重。
- 不从单张无参照照片精确反推尺寸。
- 不把不同厂商的型材、螺母和连接件默认混用。
- 不编造货号、孔位、价格、现货状态或软件文件。
- 不因横梁变形通过就写“结构安全”。
- 不在未实际打开验证时声称生成了 MayCAD、CAD 或其他专有文件。
- 不隐藏“未检查连接强度、侧向稳定、疲劳、地面固定”等边界。

## 首版支持范围

优先处理个人玩家和小工作室的货架、工作台、工具车、3D 打印机架、轻型设备底座、展示架和简单罩体。对超出范围的任务,仍可完成需求整理、结构草图、资料缺口和专业复核清单。
